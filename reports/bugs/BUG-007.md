## BUG-007: Parenthesised Expression With Trailing Multiplication Discards The Multiplier

| Field             | Detail                                                                                                     |
| ----------------- | ---------------------------------------------------------------------------------------------------------- |
| **Severity**      | Critical                                                                                                   |
| **Priority**      | P1                                                                                                         |
| **Status**        | Open                                                                                                       |
| **Type**          | Functional Defect — Parser Logic Error (Recursive Descent)                                                 |
| **Environment**   | Chromium / Firefox / WebKit — macOS / Ubuntu / https://rbihubcodechallenge.github.io/calculator/index.html |
| **Discovered By** | Automated test failures — TC-ARITH-009, TC-ARITH-010 / exploratory testing                                 |

### Preconditions

- Calculator page is loaded and display is empty.

### Steps to Reproduce

**Scenario A — Simple:**

1. Navigate to `https://rbihubcodechallenge.github.io/calculator/index.html`
2. Click **(** → **2** → **+** → **4** → **)** → **×** → **5**
3. Click **=**

**Scenario B — Nested:**

1. Clear display
2. Click **(** → **1** → **+** → **2** → **)** → **×** → **(** → **4** → **+** → **5** → **)**
3. Click **=**

### Expected Result

- Scenario A: `(2+4)×5 = 6×5 =` **`30`**
- Scenario B: `(1+2)×(4+5) = 3×9 =` **`27`**

### Actual Result

- Scenario A: **`6`** — Only `(2+4)` is evaluated; `×5` is silently discarded
- Scenario B: **`3`** — Only `(1+2)` is evaluated; `×(4+5)` is silently discarded

**Root Cause (Code Analysis):**
The obfuscated recursive descent parser contains a `parseExpression` / `parseFactor` split.
When parsing a token stream like `(2+4)*5`, the parser:

1. Recognises `(`, recurses into `parseExpression`
2. Returns `6` after matching `)`
3. Does NOT check for a continuation (`*` or `/`) after the `)` — the outer loop that handles
   `term OP factor` never re-enters after a grouped expression
4. Returns immediately, leaving `*5` unconsumed in the token stream

This means **any expression of the form `(a OP b) × c` or `(a OP b) × (c OP d)` is broken**.

```
parse("(2+4)*5") → parseFactor() → recurse → return 6
                           ↑ loop exits; *5 never seen → return 6
```

### Banking Domain Impact

This is a **showstopper defect** for any financial calculation involving grouped sub-expressions:

- **Compound interest**: `(principal × rate) × periods` — multiplier `× periods` is dropped
- **Tax calculations**: `(grossIncome − deductions) × taxRate` — tax rate discarded
- **Portfolio returns**: `(stockA + stockB) × priceMultiplier` — multiplier ignored
- **Currency conversion**: `(amount + fee) × exchangeRate` — wrong result every time

A calculator that silently drops multiplication after parentheses cannot be trusted in any
financial context. Results look plausible but are consistently wrong.

- **Automated tests (FAIL):**

  - `src/tests/functional/arithmetic.spec.ts` → TC-ARITH-009 (documents actual `"6"`)
  - `src/tests/functional/arithmetic.spec.ts` → TC-ARITH-010 (documents actual `"3"`)


### Suggested Fix

Ensure the `parseFactor` / multiplication loop re-enters after a parenthesised group:

```javascript
function parseTerm(tokens, pos) {
  let { value: left, pos: nextPos } = parseFactor(tokens, pos);
  while (nextPos < tokens.length && (tokens[nextPos] === '*' || tokens[nextPos] === '/')) {
    const op = tokens[nextPos];
    const { value: right, pos: afterRight } = parseFactor(tokens, nextPos + 1);
    left = op === '*' ? left * right : left / right;
    nextPos = afterRight;
  }
  return { value: left, pos: nextPos };
}
```
