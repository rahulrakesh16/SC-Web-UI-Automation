## BUG-006: Unbalanced Parentheses Silently Produce Incorrect Result Instead of Error

| Field             | Detail                                                                                                     |
| ----------------- | ---------------------------------------------------------------------------------------------------------- |
| **Severity**      | Medium                                                                                                     |
| **Priority**      | P2                                                                                                         |
| **Status**        | Open                                                                                                       |
| **Type**          | Functional Defect — Missing Parenthesis Validation                                                         |
| **Environment**   | Chromium / Firefox / WebKit — macOS / Ubuntu / https://rbihubcodechallenge.github.io/calculator/index.html |
| **Discovered By** | Automated regression suite — TC-BUG-006                                                                    |

### Preconditions

- Calculator page is loaded and display is empty.

### Steps to Reproduce

1. Navigate to `https://rbihubcodechallenge.github.io/calculator/index.html`
2. Click **(** — open parenthesis
3. Click **5**
4. Click **+**
5. Click **2**
6. Click **=** without entering a closing **)**

### Expected Result

Display shows **`Error`** — the expression `(5+2` is syntactically invalid. A well-formed
calculator must detect the missing closing parenthesis and reject the expression.

### Actual Result

Display shows **`7`** — the parser silently ignores the missing closing parenthesis and
computes `5+2 = 7` as if the parentheses were balanced.

**Root Cause (Code Analysis):**
The recursive descent parser does not maintain or validate a parenthesis depth counter. When
it encounters an opening `(` it begins a sub-expression, and when the token stream ends without
a matching `)` it returns the sub-expression result without signalling an error. The parser
treats end-of-input as an implicit close.

```
parse("(5+2") → parseExpression → parseTerm → matches "5+2" → returns 7
                                          ↑ never validates that ) was seen
```

### Banking Domain Impact

Undetected syntax errors in financial expressions can produce silently incorrect values:

- A user typing `(principal * rate` without closing the group expects an error, not a computed
  value based on partial expression evaluation.
- **Silent data corruption:** The result `7` is numerically plausible, making this bug extremely
  hard to detect manually — users may trust wrong results.
- **Audit trail risk:** If calculations are logged, a syntactically invalid formula producing a
  result would appear legitimate in audit logs, creating compliance risk.
- **Calculator contract violation:** A scientific calculator is expected to enforce mathematical
  notation rules strictly.

### Suggested Fix

Add parenthesis balance tracking in the parser. Before returning a result, verify depth is zero:

```javascript
function parseExpression(tokens, pos) {
  let depth = 0;
  for (const tok of tokens) {
    if (tok === '(') depth++;
    if (tok === ')') depth--;
  }
  if (depth !== 0) return 'Error';
  // ... existing parse logic
}
```
